# AGENT.md

## 1. 목적

이 문서는 커피숍 주문 시스템 프로젝트에서 AI 에이전트가 작업할 때 따라야 하는 운영 규칙을 정의한다.

이 프로젝트는 다수 서버 환경에서도 안정적으로 동작하는 포인트 기반 커피 주문 시스템을 목표로 한다.  
따라서 단순히 기능을 빠르게 구현하는 것보다, **설계 의도**, **동시성 제어**, **데이터 일관성**, **확장성**, **검증 가능성**을 명확히 유지하는 것이 중요하다.

에이전트는 모든 작업을 다음 흐름으로 진행한다.

```text
설계 → 승인 → 작업 실행 → 검증
```

---

## 2. 기본 원칙

### 2.1 설계 먼저, 구현은 나중에

에이전트는 사용자의 요청을 받으면 바로 코드를 수정하지 않는다.  
먼저 요구사항을 분석하고, 어떤 구조로 해결할지 설계안을 제시해야 한다.

특히 다음 영역에 영향을 주는 작업은 반드시 설계 단계를 거친다.

- 주문 생성
- 포인트 충전
- 포인트 차감
- 결제 처리
- 주문 이력 저장
- 인기 메뉴 집계
- 캐시 전략
- 트랜잭션 범위
- 락 전략
- DB 제약 조건
- API 응답 구조
- 예외 처리
- 테스트 구조

---

### 2.2 승인 없는 작업 실행 금지

에이전트는 설계안을 제시한 뒤 사용자의 승인을 기다린다.

사용자가 다음과 같이 명확하게 승인한 경우에만 작업을 실행한다.

- `좋아`
- `진행해줘`
- `이 방식으로 해줘`
- `수정해줘`
- `구현해줘`
- `승인`

승인이 없으면 실제 코드 수정, 파일 생성, 삭제, 리팩터링을 진행하지 않는다.

단, 사용자가 명시적으로 “바로 만들어줘”, “바로 수정해줘”, “확인해서 고쳐줘”라고 요청한 경우에는 설계 요약을 짧게 남기고 작업을 진행할 수 있다.

---

### 2.3 변경 범위 최소화

에이전트는 요청받은 범위 안에서만 작업한다.

금지되는 행동:

- 요청과 무관한 파일 수정
- 전체 구조를 임의로 갈아엎기
- 사용하지 않는 라이브러리 추가
- 기존 컨벤션과 다른 방식 도입
- 불필요한 대규모 리팩터링
- 테스트 없이 핵심 로직 변경

필요한 변경이 요청 범위를 넘어간다면 먼저 사용자에게 이유를 설명하고 승인을 받아야 한다.

---

### 2.4 이유를 설명할 수 있는 코드 작성

이 프로젝트는 설계 논리가 중요하다.  
에이전트는 단순히 “동작하는 코드”가 아니라, 왜 그런 구조를 선택했는지 설명 가능한 코드를 작성해야 한다.

특히 다음 질문에 답할 수 있어야 한다.

- 왜 이 테이블이 필요한가?
- 왜 이 API가 필요한가?
- 왜 이 트랜잭션 범위가 적절한가?
- 왜 이 락 전략을 선택했는가?
- 왜 이 데이터는 캐시해도 되는가?
- 왜 이 데이터는 반드시 DB 기준이어야 하는가?
- 다수 서버 환경에서도 안전한가?

## 3. 작업 진행 흐름

### 3.1 설계 단계

에이전트는 작업 전에 다음 내용을 정리한다.

#### 설계 단계 출력 형식

```markdown
## 작업 설계안

### 1. 요구사항 이해
- 사용자가 요청한 기능을 요약한다.

### 2. 변경 대상
- 수정 또는 생성할 파일 목록을 정리한다.

### 3. 도메인 영향
- 메뉴, 포인트, 주문, 주문상세, 인기 메뉴, Outbox 중 어떤 영역에 영향이 있는지 설명한다.

### 4. 트랜잭션/동시성 고려
- 트랜잭션이 필요한지 설명한다.
- 락이 필요한지 설명한다.
- 중복 요청 방지가 필요한지 설명한다.

### 5. 데이터 일관성 고려
- 강한 일관성이 필요한 데이터와 최종적 일관성으로 충분한 데이터를 구분한다.

### 6. 작업 순서
- 어떤 순서로 구현할지 단계별로 작성한다.

### 7. 검증 계획
- 어떤 테스트 또는 빌드 명령으로 검증할지 작성한다.
```

---

### 3.2 승인 단계

설계안을 제시한 후에는 사용자 승인을 기다린다.

승인 전에는 다음 작업을 하지 않는다.

- 코드 수정
- 파일 삭제
- DB 스키마 변경
- API 구조 변경
- 테스트 코드 변경
- 의존성 추가

단, 사용자가 설계안만 요청한 경우에는 설계안 작성으로 종료한다.

---

### 3.3 작업 실행 단계

승인이 완료되면 에이전트는 설계안에 따라 작업한다.

작업 중 지켜야 할 규칙:

- 기존 프로젝트 구조와 패키지 컨벤션을 따른다.
- 하나의 작업 단위는 가능한 작게 유지한다.
- 핵심 도메인 로직은 Service 계층에 둔다.
- Controller는 요청 검증과 응답 반환에 집중한다.
- Entity를 API 응답으로 직접 반환하지 않는다.
- Request/Response DTO를 사용한다.
- 트랜잭션 경계는 Service 계층에서 관리한다.
- 포인트 차감, 주문 생성, 이력 저장은 하나의 트랜잭션 안에서 처리한다.
- 다수 서버 환경에서 깨지는 인메모리 상태 저장은 사용하지 않는다.

---

### 3.4 검증 단계

작업이 끝나면 반드시 검증 결과를 정리한다.

#### 검증 단계 출력 형식

```markdown
## 작업 완료 보고

### 1. 변경 요약
- 어떤 기능을 구현/수정했는지 요약한다.

### 2. 변경 파일
- 수정 또는 생성한 파일 목록을 작성한다.

### 3. 검증 결과
- 실행한 테스트 또는 빌드 명령을 적는다.
- 성공/실패 여부를 명확히 적는다.

### 4. 주의할 점
- 남은 위험, 추가 확인이 필요한 부분, 운영 시 고려할 점을 적는다.
```

테스트나 빌드를 실행하지 못한 경우에는 실행하지 못했다고 솔직하게 적고, 사용자가 직접 실행할 명령어를 제공한다.

예시:

```bash
./gradlew clean build
./gradlew test
```

Windows 환경이라면 다음 명령어를 우선 안내한다.

```bash
./gradlew.bat clean build
./gradlew.bat test
```

## 4. 도메인 설계 기준

### 4.1 메뉴

메뉴는 주문 가능한 커피 상품 정보를 의미한다.

기본 규칙:

- 메뉴 목록 조회는 읽기 전용 API이다.
- 판매 중인 메뉴만 주문할 수 있다.
- 주문 시 클라이언트가 보낸 가격은 신뢰하지 않는다.
- 주문 시점의 가격은 반드시 DB의 메뉴 가격을 기준으로 계산한다.
- 메뉴 목록은 캐시할 수 있지만, 결제 금액 계산은 캐시가 아닌 DB 기준으로 수행한다.

---

### 4.2 포인트

포인트는 커피 주문에 사용되는 결제 수단이다.

기본 규칙:

- 커피 주문은 포인트로만 가능하다.
- 포인트 잔액은 `user_point` 기준으로 판단한다.
- 포인트 충전과 차감은 반드시 이력으로 남긴다.
- 포인트 차감 시 잔액이 음수가 되면 안 된다.
- 다수 서버 환경에서도 안전해야 하므로 API 서버 메모리 기반 락을 사용하지 않는다.
- 포인트 변경은 DB 트랜잭션 안에서 처리한다.

권장 전략:

```text
user_point row lock 획득
→ 잔액 검증
→ 포인트 변경
→ point_history 저장
→ transaction commit
```

---

### 4.3 주문/결제

주문과 결제는 하나의 원자적 흐름으로 처리한다.

기본 규칙:

- 주문 생성과 포인트 차감은 하나의 트랜잭션으로 묶는다.
- 포인트 부족 시 주문은 생성하지 않는다.
- 주문 생성 성공 후 포인트 차감 실패 상태를 허용하지 않는다.
- 포인트 차감 성공 후 주문 생성 실패 상태를 허용하지 않는다.
- 주문 당시 메뉴명과 가격은 `order_item`에 snapshot으로 저장한다.
- 같은 주문 요청이 중복 처리되지 않도록 `requestId`를 사용한다.

권장 처리 순서:

```text
requestId 중복 확인
→ 메뉴 유효성 검증
→ DB 기준 가격 계산
→ user_point row lock 획득
→ 잔액 검증
→ 포인트 차감
→ orders 저장
→ order_item 저장
→ point_history 저장
→ order_outbox 저장
→ transaction commit
```

---

### 4.4 인기 메뉴

인기 메뉴는 주문 내역을 기반으로 추천한다.

기본 규칙:

- 인기 메뉴는 주문/결제만큼 강한 일관성이 필요하지 않다.
- 인기 메뉴 조회는 성능이 중요하다.
- 매 요청마다 원본 주문 테이블을 직접 `group by` 하지 않는다.
- 집계 테이블 또는 Redis 캐시를 활용한다.
- 주문 성공 이벤트는 유실되면 안 되므로 Outbox 패턴을 사용할 수 있다.

권장 구조:

```text
주문 완료
→ order_outbox 저장
→ Worker가 이벤트 처리
→ menu_order_stat 갱신
→ Redis 캐시 갱신 또는 만료
→ 인기 메뉴 API 응답
```

## 5. API 설계 기준

필수 API는 다음 4개이다.

| 기능 | Method | Endpoint | 설명 |
|---|---:|---|---|
| 커피 메뉴 목록 조회 | GET | `/api/menus` | 판매 중인 커피 메뉴 목록 조회 |
| 포인트 충전하기 | POST | `/api/points/charge` | 사용자 포인트 충전 |
| 커피 주문, 결제하기 | POST | `/api/orders` | 포인트 차감과 주문 생성 |
| 인기 메뉴 목록 조회 | GET | `/api/menus/popular` | 주문 내역 기반 인기 메뉴 조회 |

API 설계 규칙:

- Controller는 Entity를 직접 반환하지 않는다.
- Request DTO와 Response DTO를 분리한다.
- 요청 값 검증을 명확히 한다.
- 실패 케이스를 명확한 예외로 처리한다.
- 중복 요청이 가능한 API는 `requestId`를 사용한다.
- 주문/결제 API는 클라이언트 가격을 신뢰하지 않는다.
- 인기 메뉴 API는 `period`, `limit` 같은 확장 가능한 query parameter를 고려한다.

## 6. DB 설계 기준

### 6.1 주요 테이블

권장 테이블:

- `menu`
- `user_point`
- `point_history`
- `orders`
- `order_item`
- `order_outbox`
- `menu_order_stat`

---

### 6.2 단방향 연관관계 원칙

이 프로젝트의 테이블 연관관계는 단방향 구조를 우선한다.

기본 원칙:

- FK를 가진 테이블에서 대상 테이블로만 참조한다.
- 양방향 연관관계는 기본적으로 만들지 않는다.
- 필요 없는 컬렉션 필드는 Entity에 추가하지 않는다.
- `user_id`는 별도 User Entity 없이 스칼라 값으로 관리할 수 있다.
- `order_outbox`는 논리적 이벤트 저장 테이블로 FK 없이 운영할 수 있다.

권장 관계:

```text
order_item.order_id        → orders.id
order_item.menu_id         → menu.id
menu_order_stat.menu_id    → menu.id
```

주의:

```text
orders → order_item 컬렉션 양방향 매핑은 기본적으로 만들지 않는다.
menu → order_item 컬렉션 양방향 매핑은 기본적으로 만들지 않는다.
user_point → point_history 관계를 억지로 연결하지 않는다.
order_outbox에 불필요한 FK를 만들지 않는다.
```

---

### 6.3 제약 조건

권장 제약 조건:

```text
orders: unique(user_id, request_id)
point_history: unique(user_id, request_id, type)
menu_order_stat: unique(menu_id, stat_date)
user_point: unique(user_id)
```

목적:

- 중복 주문 방지
- 중복 충전 방지
- 사용자별 포인트 row 단일화
- 메뉴별 일자별 인기 메뉴 집계 중복 방지

---

### 6.4 인덱스

권장 인덱스:

```text
menu(status)
orders(user_id, created_at)
order_item(menu_id)
menu_order_stat(stat_date, total_quantity)
order_outbox(status, created_at)
```

목적:

- 판매 중 메뉴 조회 최적화
- 사용자 주문 내역 조회 확장 대비
- 인기 메뉴 집계 조회 최적화
- Outbox Worker 처리 대상 조회 최적화

## 7. 동시성 처리 기준

### 7.1 포인트 차감

포인트 차감은 반드시 동시성 문제를 고려한다.

사용자가 동시에 여러 주문을 보내도 잔액이 음수가 되면 안 된다.

권장 방식:

- `user_point` row에 비관적 락 적용
- 잔액 검증 후 차감
- 차감 이력 저장
- 트랜잭션 commit

금지 방식:

- API 서버 인메모리 락
- 단순 조회 후 업데이트
- 잔액 검증 없이 차감
- Redis만 최종 포인트 저장소로 사용하는 방식

---

### 7.2 중복 요청

네트워크 재시도, 버튼 중복 클릭, 클라이언트 타임아웃 때문에 같은 요청이 여러 번 들어올 수 있다.

권장 방식:

- 주문/충전 요청에 `requestId` 포함
- DB unique constraint로 중복 방지
- 같은 `requestId` 재요청 시 기존 결과 반환

---

### 7.3 Outbox 이벤트 처리

주문 성공 후 인기 메뉴 집계 이벤트가 유실되면 안 된다.

권장 방식:

- 주문 트랜잭션 안에서 `order_outbox` 저장
- Worker가 `READY` 이벤트를 조회
- 집계 성공 시 `DONE`
- 실패 시 `retry_count` 증가
- 일정 횟수 이상 실패 시 `FAILED`

---

## 8. 코드 작성 기준

### 8.1 계층 구조

권장 계층:

```text
controller
service
repository
domain/entity
dto/request
dto/response
exception
```

역할:

- Controller: 요청 검증, 인증 사용자 식별, Service 호출, 응답 반환
- Service: 비즈니스 로직, 트랜잭션, 동시성 제어
- Repository: DB 접근
- Entity/Domain: 상태와 도메인 행위 표현
- DTO: API 요청/응답 모델
- Exception: 도메인 예외와 에러 코드 관리

---

### 8.2 트랜잭션

기본 규칙:

- 조회 전용 메서드는 `readOnly = true`를 고려한다.
- 상태 변경 메서드는 일반 트랜잭션을 사용한다.
- 주문/결제 로직은 하나의 트랜잭션으로 묶는다.
- 트랜잭션 안에서 외부 API 호출을 길게 수행하지 않는다.
- 락을 잡는 구간은 가능한 짧게 유지한다.

---

### 8.3 예외 처리

명확히 구분해야 하는 예외:

- 존재하지 않는 메뉴
- 판매 중지된 메뉴
- 수량 오류
- 포인트 부족
- 중복 요청
- 존재하지 않는 사용자 포인트
- 주문 조회 실패
- Outbox 처리 실패

예외 메시지는 사용자에게 필요한 정보만 제공하고, 내부 구현 세부사항은 노출하지 않는다.

## 9. 테스트 기준

### 9.1 필수 테스트

다음 케이스는 우선적으로 테스트한다.

#### 메뉴

- 판매 중 메뉴 목록 조회 성공
- 판매 중지 메뉴는 주문 불가

#### 포인트

- 포인트 충전 성공
- 포인트 충전 중복 요청 시 중복 반영 방지
- 포인트 차감 성공
- 포인트 부족 시 실패
- 동시에 주문 요청 시 잔액 음수 방지

#### 주문

- 주문 성공
- 주문 시 DB 기준 가격 사용
- 주문 당시 메뉴명/가격 snapshot 저장
- 존재하지 않는 메뉴 주문 실패
- 수량 0 이하 실패
- 같은 requestId 주문 재요청 시 기존 결과 반환

#### 인기 메뉴

- 주문 완료 후 outbox 저장
- outbox 처리 후 menu_order_stat 갱신
- 인기 메뉴 조회 성공
- Redis cache hit/miss 흐름 검증

## 11. 금지사항

에이전트는 다음 작업을 임의로 수행하지 않는다.

- 승인 없는 대규모 리팩터링
- 승인 없는 DB 구조 변경
- 승인 없는 의존성 추가
- 승인 없는 패키지 구조 변경
- 기존 테스트 삭제
- 예외를 숨기는 try-catch 남용
- 동시성 문제를 무시한 포인트 차감 구현
- 주문/결제 로직을 여러 트랜잭션으로 쪼개기
- API 서버 메모리에 주문 상태 또는 중복 요청 상태 저장
- 클라이언트가 보낸 가격으로 결제 금액 계산
- 인기 메뉴 조회 시 매번 전체 주문 테이블을 무조건 full scan 하는 구조

## 12. 좋은 작업 예시

### 12.1 좋은 설계 예시

```markdown
포인트 차감은 user_point row에 비관적 락을 적용합니다.
이유는 포인트가 금전성 데이터이고, 동시에 여러 주문 요청이 들어와도 잔액이 음수가 되면 안 되기 때문입니다.
락 범위는 사용자 한 명의 user_point row로 제한하여 전체 시스템 성능 저하를 줄입니다.
```

---

### 12.2 나쁜 설계 예시

```markdown
포인트 잔액을 조회한 뒤 차감해서 저장합니다.
동시에 요청이 들어오면 나중에 처리하면 됩니다.
```

문제:

- 동시에 두 요청이 같은 잔액을 읽을 수 있다.
- 잔액이 음수가 될 수 있다.
- 다수 서버 환경에서 데이터 정합성이 깨질 수 있다.

## 13. 최종 기준

이 프로젝트에서 에이전트의 역할은 단순 코드 생성이 아니다.

에이전트는 다음을 함께 보장해야 한다.

- 요구사항 충족
- 설계 의도 유지
- 동시성 안전성
- 데이터 일관성
- 확장 가능성
- 테스트 가능성
- 사용자 승인 기반 작업 진행

따라서 모든 작업은 다음 흐름을 따른다.

```text
설계 → 승인 → 작업 실행 → 검증
```
