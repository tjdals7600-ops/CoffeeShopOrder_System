# 커피숍 주문 시스템 트러블슈팅

이 문서는 커피숍 주문 시스템을 구현하면서 발생할 수 있는 주요 문제와 해결 방식을 정리한 기록입니다.
현재 프로젝트의 포인트 충전, 주문/결제, 인기 메뉴 조회, Redis 캐시 구조를 기준으로 작성했습니다.

---

## 1. 포인트 동시 차감으로 잔액이 틀어지는 문제

### 문제 상황

사용자가 10,000P를 가지고 있고, 동시에 여러 주문 요청이 들어오면 각 요청이 같은 잔액을 읽고 결제를 진행할 수 있습니다.

예를 들어 8,000P 주문 2건이 거의 동시에 들어오면, 두 요청이 모두 결제 전 잔액을 10,000P로 보고 성공 처리할 위험이 있습니다.
이 경우 실제 보유 포인트보다 많은 금액이 결제되는 데이터 정합성 문제가 발생합니다.

### 원인

포인트는 금액성 데이터이기 때문에 읽기와 쓰기 사이에 경쟁 조건이 생기면 안 됩니다.
하지만 단순히 `findByUserId`로 잔액을 조회한 뒤 차감하면, 여러 트랜잭션이 같은 값을 동시에 읽을 수 있습니다.

### 해결 방식

`UserPointRepository`에서 비관적 락을 사용하도록 `findByUserIdForUpdate`를 만들었습니다.
포인트 충전과 주문 결제에서는 이 메서드로 `user_point` row를 잠근 뒤 잔액을 변경합니다.

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select up from UserPoint up where up.userId = :userId")
Optional<UserPoint> findByUserIdForUpdate(@Param("userId") Long userId);
```

적용 흐름은 다음과 같습니다.

```text
1. 주문 요청 검증
2. 주문 메뉴와 총 금액 계산
3. user_point row lock 획득
4. 최신 잔액 기준으로 결제 가능 여부 확인
5. 포인트 차감
6. 주문, 주문 항목, 포인트 이력 저장
7. 트랜잭션 commit
```

### 검증

주문 테스트에서 결제 성공 후 잔액, 주문 수, 주문 항목 수, 포인트 이력 수가 정확한지 검증했습니다.
잔액 부족 케이스에서는 주문과 outbox가 생성되지 않는 것도 확인했습니다.

## 2. 같은 요청이 반복되어 중복 충전/중복 결제되는 문제

### 문제 상황

클라이언트가 네트워크 지연, 타임아웃, 버튼 중복 클릭 때문에 같은 요청을 다시 보낼 수 있습니다.

포인트 충전 요청이 중복 처리되면 실제보다 포인트가 많이 충전됩니다.
주문 요청이 중복 처리되면 포인트가 두 번 차감되고 주문도 두 건 생성됩니다.

### 원인

HTTP 요청은 클라이언트 입장에서 응답을 받지 못했다고 해서 서버 처리가 실패했다고 보장할 수 없습니다.
서버가 이미 처리한 요청을 다시 새 요청처럼 처리하면 중복 반영 문제가 발생합니다.

### 해결 방식

요청마다 `requestId`를 받도록 하고, 사용자 식별값과 함께 멱등성 기준으로 사용했습니다.

주문은 `orders` 테이블의 `user_id + request_id` 조합으로 중복 요청을 판단합니다.
포인트 충전은 `point_history`에서 `user_id + request_id + type` 기준으로 이미 처리된 충전 이력을 조회합니다.

서비스에서는 트랜잭션 시작 직후 한 번 확인하고, 포인트 row lock을 획득한 뒤 한 번 더 확인합니다.
이렇게 하면 거의 동시에 들어온 같은 요청도 중복 처리될 가능성을 줄일 수 있습니다.

```text
1. requestId로 기존 처리 결과 조회
2. 기존 결과가 있으면 새로 처리하지 않고 기존 응답 반환
3. 기존 결과가 없으면 user_point row lock 획득
4. lock 이후 다시 기존 결과 조회
5. 여전히 없을 때만 충전 또는 주문 처리
```

### 검증

테스트 코드에서 같은 `requestId`로 두 번 요청했을 때 다음을 확인했습니다.

- 포인트 충전은 잔액과 이력이 한 번만 반영됩니다.
- 주문 결제는 주문, 주문 항목, 포인트 이력, outbox가 한 번만 생성됩니다.
- 두 번째 요청은 실패가 아니라 기존 성공 결과를 반환합니다.

## 3. 인기 메뉴 캐시가 최신 주문 집계와 어긋나는 문제

### 문제 상황

인기 메뉴 API는 최근 7일간 인기 있는 메뉴 3개를 빠르게 조회해야 합니다.
하지만 매 요청마다 주문 원장을 직접 집계하면 주문 데이터가 많아질수록 조회 성능이 떨어집니다.

반대로 Redis 캐시만 사용하면 주문이 새로 발생한 뒤에도 오래된 인기 메뉴 결과가 반환될 수 있습니다.

### 원인

인기 메뉴 조회는 읽기 성능이 중요하지만, 주문 성공 이후 집계 결과와 캐시가 어느 시점에 갱신되는지도 중요합니다.
특히 주문 트랜잭션이 rollback 되었는데 캐시가 먼저 비워지거나, commit 되기 전에 외부 반영이 일어나면 일관성이 흔들릴 수 있습니다.

### 해결 방식

인기 메뉴는 주문 원장을 직접 조회하지 않고 `menu_order_stat` 집계 테이블을 기준으로 조회하도록 했습니다.
주문 성공 시 주문 항목을 기준으로 메뉴별 일자 집계를 누적합니다.

조회 성능을 위해 `MenuService#getPopularMenus`에 Redis 캐시를 적용했습니다.
캐시 무효화는 `CacheInvalidationService`에서 트랜잭션 commit 이후 실행되도록 처리했습니다.

```text
1. 주문/결제 트랜잭션 시작
2. 주문 항목 저장
3. menu_order_stat 집계 누적
4. 트랜잭션 commit
5. afterCommit에서 인기 메뉴 캐시 삭제
6. 다음 인기 메뉴 조회 시 집계 테이블 기준으로 새 결과 캐싱
```

테스트 환경에서는 외부 Redis 없이 검증할 수 있도록 `application-test.yml`에서 simple cache를 사용합니다.

```yaml
spring:
  cache:
    type: simple
```

### 검증

인기 메뉴 테스트에서 다음을 확인했습니다.

- 기본 조회는 `WEEKLY`, `limit=3` 기준으로 동작합니다.
- 최근 7일 범위 밖의 집계는 순위에 포함되지 않습니다.
- 메뉴별 `orderCount`와 `totalQuantity`가 정확히 합산됩니다.
- 반복 조회 시 캐시가 적용됩니다.
- 주문 성공 후 인기 메뉴 캐시가 commit 이후 무효화됩니다.

---

## 정리

이번 프로젝트에서 핵심 문제는 모두 데이터 정합성과 조회 성능 사이의 균형이었습니다.

포인트와 주문처럼 틀리면 안 되는 데이터는 DB 트랜잭션, row lock, 멱등성 키를 사용했습니다.
인기 메뉴처럼 빠르게 조회해야 하는 데이터는 집계 테이블과 Redis 캐시를 사용하되, commit 이후 캐시를 무효화해 일관성을 맞췄습니다.
